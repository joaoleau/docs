##### Deduplicação de Mensagens e Parâmetros

`**MessageGroupId**`: garante a **ordem de processamento** dentro de grupos de mensagens, mas não evita duplicatas.

`**ReceiveRequestAttemptId**`: evita duplicação de chamadas **de leitura**, não de envio.

Resposta:
- **MessageDeduplicationId**: é o identificador utilizado para a deduplicação das mensagens enviadas. ### Redis

##### Modo Cluster
- O ElastiCache para Redis com Modo Cluster habilitado oferece alta confiabilidade, disponibilidade e escalabilidade horizontal, permitindo distribuir dados em múltiplos shards. Isso reduz o impacto no desempenho e facilita a manutenção do cache, especialmente quando a regeneração do conteúdo é custosa.
- **Sincronização primário→réplicas**: réplicas fazem **replicação assíncrona**/semisíncrona do primário; durante failover réplicas podem ser promovidas a primário.
- **Primário (primary/master)**: aceita gravações.
- **Réplica (replica/read-only)**: atende leituras, replica dados do primário.

- **Cluster Mode Disabled**: um primário (writer) com 0..N réplicas (read replicas). Escala leitura adicionando réplicas; failover automático para réplica se habilitado.
- **Cluster Mode Enabled (sharding)**: dados particionados em múltiplos shards (node groups). Cada shard tem um primário e 0..N réplicas. Permite partição horizontal (mais capacidade e memória agregada) e redimensionamento de shards online.
### Estratégias de Cache

##### Write-through
A equipe de desenvolvimento de uma empresa de comércio eletrônico está se preparando para a próxima promoção de Ação de Graças. O gerente de produto deseja que a equipe implemente uma estratégia de cache adequada no Amazon ElastiCache para suportar picos de tráfego no site durante a promoção. Um requisito fundamental é garantir atualizações consistentes nos preços e descrições dos produtos, para que **o cache nunca fique dessincronizado com o backend**. Como um Desenvolvedor Associado, qual das seguintes soluções você recomendaria para esse caso de uso?

~={red}Utilizar uma estratégia de cache que escreva diretamente no cache e sincronize o backend posteriormente: =~
Incorreta. Essa abordagem não é adequada para o caso, pois os preços e descrições dos produtos precisam estar sempre consistentes com o backend. Sincronizar o backend posteriormente pode causar inconsistências temporárias inaceitáveis.

~={red}Utilizar uma estratégia de cache que atualize o cache e o backend simultaneamente.=~
Incorreta. Cache e backend são sistemas distintos e não podem ser atualizados simultaneamente em uma única operação atômica, tornando essa opção inviável para garantir consistência imediata.

~={green}Utilizar uma estratégia de cache que escreva primeiro no backend e, em seguida, invalide o cache.=~
Correta. Essa estratégia, semelhante ao write-through, garante que o backend seja atualizado primeiro e depois o cache seja invalidado, forçando a atualização do cache com os dados mais recentes, mantendo a consistência entre cache e backend.

> **Write-through:** Estratégia de cache em que a **atualização é gravada primeiro no backend** (banco de dados) e o cache é **atualizado ou invalidado logo em seguida**, garantindo consistência imediata
> 	**Invalidação de cache:** Processo de **remover ou marcar como expirado** um item armazenado, forçando sua atualização na próxima leitura.
> 	Essa abordagem garante que o cache **nunca sirva informações obsoletas**, o que é crucial para dados sensíveis como **preços, saldos ou descrições de produtos**.

##### Lazy Loading
Você está criando uma aplicação web na qual os usuários podem seguir uns aos outros. Alguns usuários serão mais populares que outros e, portanto, seus dados serão solicitados com muita frequência. ~={green}Utilizar a estratégia Lazy Loading com TTL=~
Correta. **A estratégia Lazy Loading carrega os dados no cache somente quando são solicitados, garantindo que apenas os perfis mais acessados sejam armazenados, economizando recursos. O uso do TTL permite que os dados expirem automaticamente após um minuto, mantendo a desatualização dentro do limite aceitável para o sistema de alto tráfego descrito.**

**Lazy Loading:** Estratégia de cache que armazena dados apenas quando são solicitados pela primeira vez, evitando o cache de informações pouco utilizadas.

##### ECS e CloudWatch Logs

~={red}{ ""Version"": ""2012-10-17"", ""Statement"": [ { ""Effect"": ""Allow"", ""Action"": [ ""logs:CreateLogGroup"", ""logs:CreateLogStream"", ""logs:PutLogEvents"" ], ""Resource"": [ ""arn:aws:logs:*:*:*"" ] } ] }=~

##### Métrica de Backlog com política de escalonamento por target tracking
Uma empresa de comércio eletrônico desenvolveu uma API hospedada no Amazon ECS. **Picos variáveis de tráfego na aplicação estão causando atrasos no processamento dos pedidos.** A aplicação **processa os pedidos utilizando filas do Amazon SQS**. A métrica ApproximateNumberOfMessagesVisible apresenta picos muito altos ao longo do dia, o que dispara o alarme do CloudWatch. Outras métricas do ECS para os containers da API estão dentro dos limites esperados. Como um Desenvolvedor Associado, qual das seguintes opções você recomendaria para melhorar o desempenho mantendo os custos baixos?

~={green}Utilizar métrica de backlog por instância com política de escalonamento por rastreamento de alvo (target tracking)=~
Correta. **Utilizar uma métrica personalizada de backlog por instância com uma política de escalonamento por rastreamento de alvo permite que o sistema ajuste dinamicamente a capacidade do ECS com base na quantidade de mensagens pendentes por instância**, garantindo que o processamento acompanhe a demanda de forma eficiente e mantendo os custos controlados. Essa abordagem é mais precisa do que usar apenas a métrica ApproximateNumberOfMessagesVisible, pois considera a capacidade atual da frota e o tempo aceitável de latência para processar mensagens.

> Objetivo: escalar o número de tasks/instâncias do ECS baseado na carga real da fila, não só em CPU/Memory.
> Métrica de backlog por instância = (número de mensagens visíveis na fila) / (número de tasks ativas). A política de target‑tracking mantém essa razão próxima a um valor alvo (ex.: 50 mensagens por task).
> Benefício: quando o backlog por task sobe, escala-se automaticamente; quando cai, reduz‑se — bom para picos variáveis mantendo custos controlados.

##### ALB (X-Forwarded-For) X Network LB 
Você está projetando uma aplicação de alto desempenho que requer milhões de conexões simultâneas. Você possui várias instâncias EC2 executando servidores web Apache2 e a aplicação **precisa capturar o endereço IP de origem e a porta de origem do usuário sem utilizar o cabeçalho X-Forwarded-For.** Qual das seguintes opções atenderá a essa necessidade?

~={red}Application Load Balancer=~
Incorreta. O Application Load Balancer opera na camada 7 (aplicação) do modelo OSI e é projetado para roteamento baseado em conteúdo, como URLs e cabeçalhos HTTP. **Ele não preserva diretamente o endereço IP e a porta de origem do cliente, dependendo do cabeçalho X-Forwarded-For para essa informação, o que não atende ao requisito da pergunta.**

~={green}Network Load Balancer=~
Correta. O Network Load Balancer opera na camada 4 (transporte) do modelo OSI e é capaz de lidar com milhões de conexões por segundo. Ele mantém as conexões TCP intactas, permitindo que o endereço IP e a porta de origem do cliente sejam preservados sem a necessidade do cabeçalho X-Forwarded-For, ideal para aplicações que exigem alta performance e rastreamento da origem do tráfego.

##### Configuração de Query String X Redirecionamento

Incorreta. Esta configuração representa uma ação de redirecionamento, não uma condição de query string. Ela redireciona requisições HTTP para HTTPS mantendo o host, caminho e query string originais, mas não define uma condição para filtragem baseada em query strings.

##### SSE-C X SSE-KMS X SSE-S3 X Client-Side Encryption
Como parte das normas internas, você deve garantir que todas as comunicações com o Amazon S3 sejam criptografadas. Para qual dos seguintes **mecanismos de criptografia uma requisição será rejeitada caso a conexão não utilize HTTPS**?

~={red}SSE-KMS (Server-Side Encryption with AWS KMS Keys)=~
Incorreta. A criptografia do lado servidor usando chaves gerenciadas pelo AWS KMS não exige que a conexão seja via HTTPS para que a requisição seja aceita. Embora o uso de HTTPS seja uma boa prática, o Amazon S3 não rejeita requisições HTTP com SSE-KMS.

~={green}SSE-C (Server-Side Encryption with Customer-Provided Keys)=~
Correta. A criptografia do lado servidor com chaves fornecidas pelo cliente (SSE-C) exige que a chave de criptografia seja enviada junto com a requisição. O Amazon S3 rejeita qualquer requisição feita via HTTP (não segura) para proteger a chave, pois o envio da chave por HTTP pode comprometer sua segurança. Portanto, o uso de HTTPS é obrigatório para SSE-C.

SSE‑C (Server‑Side Encryption with Customer‑Provided Keys)

O que é: 
- servidor S3 cifra/des cifra objetos usando uma chave AES-256 que você fornece em cada requisição; a AWS não armazena essa chave.
- Como funciona (técnico): você envia a chave por cabeçalhos na requisição (algoritmo + chave base64 + MD5 da chave). S3 usa essa chave para a operação e descarta a cópia da chave; sem a chave, o objeto fica inacessível.
- Cabeçalhos usados: `x-amz-server-side-encryption-customer-algorithm = AES256`, `x-amz-server-side-encryption-customer-key (base64)`, `x-amz-server-side-encryption-customer-key-MD5 (base64)`.

```
# gerar chave 32 bytes
openssl rand -out key.bin 32

# upload com SSE-C (AWS CLI calcula o MD5 automaticamente)
aws s3api put-object --bucket meu-bucket --key arquivo.txt --body arquivo.txt --sse-c AES256 --sse-c-key fileb://key.bin

# download com SSE-C
aws s3api get-object --bucket meu-bucket --key arquivo.txt arquivo-baixado.txt --sse-c AES256 --sse-c-key fileb://key.bin
```



~={red}SSE-S3 (Server-Side Encryption with Amazon S3-Managed Keys)=~
Incorreta. A criptografia do lado servidor com chaves gerenciadas pelo próprio Amazon S3 (SSE-S3) não exige que a conexão seja via HTTPS para aceitar a requisição. O Amazon S3 criptografa os dados em repouso, mas não rejeita requisições HTTP.

~={red}Criptografia do Lado Cliente (Client-Side Encryption)=~
Incorreta. Na criptografia do lado cliente, os dados são criptografados antes de serem enviados para o Amazon S3, e a comunicação pode ocorrer via HTTP ou HTTPS. Não há obrigatoriedade do uso de HTTPS para que a requisição seja aceita, embora seja recomendável para segurança da transmissão.

### Dynamo

##### WCU
Você é um desenvolvedor DynamoDB em uma empresa aeroespacial que precisa gravar 6 objetos por segundo, cada um com tamanho de 4,5 KB. Qual é a unidade de capacidade de gravação (Write Capacity Unit - WCU) necessária para o seu projeto?

~={green}30=~
Correta. **Cada unidade de capacidade de gravação (WCU) representa uma gravação por segundo para um item de até 1 KB**. Como o tamanho do item é 4,5 KB, ele é arredondado para 5 KB, consumindo 5 WCUs por gravação. Para 6 gravações por segundo, o total é 6 x 5 = 30 WCUs

##### Hot-partition
Sua aplicação web lê e grava dados em uma tabela DynamoDB. A tabela está provisionada com 400 Unidades de Capacidade de Gravação (WCU) compartilhadas entre 4 partições. Uma das partições recebe 250 WCU/segundo, enquanto as outras recebem muito menos. Você recebe o erro **'ProvisionedThroughputExceededException'**. Qual é a causa mais provável desse erro?

~={green}Você tem uma partição quente (hot partition).=~
Correta. Nem sempre é possível distribuir a atividade de leitura e gravação de forma uniforme entre as partições. **Quando o acesso aos dados é desequilibrado, uma ""partição quente"" recebe um volume maior de tráfego de leitura e gravação em comparação às outras.** O DynamoDB utiliza a capacidade adaptativa para permitir que sua aplicação continue lendo e gravando nessas partições quentes sem ser limitada, **desde que o tráfego não ultrapasse a capacidade total provisionada da tabela ou o limite máximo da partição.**

> **ProvisionedThroughputExceededException:** Erro lançado pelo **Amazon DynamoDB** quando o volume de leituras ou gravações ultrapassa a **capacidade provisionada** de uma ou mais partições da tabela.

DynamoDB divide a tabela em partições físicas. Cada partição tem:
- um limite de armazenamento e capacidade (historicamente ~10 GB por partição),
- uma parcela da capacidade provisionada (WCU/RCU) da tabela.
- O mapeamento é feito pelo hash do partition key (valor da chave): valores com hashes que caem na mesma faixa vão para a mesma partição.
- O número de partições é determinado automaticamente pela AWS com base no tamanho da tabela e na capacidade provisionada (ex.: se provisionar muita capacidade ou tabela grande, mais partições são criadas).
- Várias `partition key` diferentes (com hashes que caem na mesma faixa) vão compartilhar a mesma partição física e competirão pela mesma parcela de capacidade (WCU/RCU) dessa partição.
- partição física = unidade de armazenamento/capacidade interna do DynamoDB (onde dados e WCU/RCU são realmente alocados).

~={red}As Unidades de Capacidade de Gravação (WCU) são aplicadas a todas as suas tabelas DynamoDB e precisam ser reconfiguradas.=~
Incorreta. As Unidades de Capacidade de Leitura (RCU) e Gravação (WCU) são configuradas de forma independente para cada tabela DynamoDB. Não existe compartilhamento de capacidade entre tabelas, portanto essa afirmação está incorreta.

##### KCL em instâncias EC2
Um dos seus Kinesis Streams está passando por um aumento de tráfego devido a um dia de promoção. Por isso, seu administrador de Kinesis realizou um split de shards, passando de 6 shards para 10 shards no seu Kinesis Stream. Sua aplicação consumidora está rodando uma aplicação baseada em KCL em instâncias EC2. Qual é o número máximo de instâncias EC2 que podem ser implantadas para processar os shards?

~={green}10=~
Correta. **Cada shard em um Kinesis Data Stream é processado por um único worker da aplicação KCL.** Como cada instância EC2 executa um worker, o número máximo de instâncias EC2 que podem processar os shards é igual ao número de shards, ou seja, 10.

**KCL (Kinesis Client Library):** Biblioteca que facilita o desenvolvimento de aplicações consumidoras de Kinesis, gerenciando automaticamente **balanceamento de carga, checkpointing** e **recuperação de falhas** entre os workers.

Em um **Kinesis Data Stream**, cada **shard** é processado por um único **worker** da aplicação KCL. Por padrão, o KCL distribui dinamicamente os shards entre os workers disponíveis, garantindo que **cada shard seja processado por apenas um worker ativo por vez**, mantendo a **ordem e a consistência** dos dados dentro daquele shard.

##### Kinesis Agents
Uma empresa possui várias instâncias **EC2 baseadas em Linux que geram diversos arquivos de log que precisam ser analisados para fins de segurança e conformidade.** A empresa deseja **usar o Kinesis Data Streams (KDS) para analisar esses dados de log**. Qual das seguintes opções é a forma mais otimizada de **enviar os dados de log das instâncias EC2 para o KDS**?

~={green}Instalar e configurar o Kinesis Agent em cada uma das instâncias=~
Correta. O Kinesis Agent é um aplicativo Java independente que oferece uma maneira simples e eficiente de **coletar e enviar dados para o Kinesis Data Streams. Ele monitora continuamente um conjunto de arquivos, enviando novos dados para o stream, gerenciando rotação de arquivos, checkpoints e tentativas em caso de falhas.** Além disso, **emite métricas no Amazon CloudWatch para monitoramento e solução de problemas**, tornando-o ideal para ambientes Linux como servidores web, de logs e bancos de dados.

~={red}Usar a Kinesis Producer Library (KPL) para coletar e ingerir dados de cada instância EC2=~
Incorreta. **A Kinesis Producer Library é uma biblioteca configurável que facilita a escrita em streams do Kinesis, atuando como intermediária entre o código do produtor e as APIs do Kinesis.** No entanto, ela não é otimizada para monitorar arquivos de log continuamente, função que o Kinesis Agent realiza de forma mais direta e eficiente.
● **Kinesis Producer Library (KPL):** biblioteca otimizada para produção de registros em streams Kinesis, oferecendo buffering e compactação, mas **não monitora arquivos de log automaticamente**.
##### Kinesis Data Stream Enhanced Fan-Out
Uma empresa de análise está utilizando o Kinesis Data Streams (KDS) para processar dados de status de saúde de automóveis de táxis gerenciados por um serviço de transporte por aplicativo. Múltiplas aplicações consumidoras estão utilizando os fluxos de dados recebidos, e os engenheiros notaram uma lentidão no desempenho da entrega dos dados entre os produtores e consumidores dos fluxos de dados. Como um Desenvolvedor Associado, qual das seguintes opções você sugeriria para melhorar o desempenho para esse caso de uso?

~={green}Utilizar o recurso Enhanced Fan-Out do Kinesis Data Streams=~
Correta. **Por padrão, a taxa de saída de 2 MB/segundo por shard é compartilhada entre todas as aplicações consumidoras**, o que pode causar gargalos quando há múltiplos consumidores. **O recurso Enhanced Fan-Out permite que cada consumidor registrado receba seu próprio throughput dedicado de 2 MB/segundo por shard**, escalando automaticamente conforme o número de shards, melhorando significativamente o desempenho para múltiplos consumidores simultâneos.

~={red}Substituir o Kinesis Data Streams pelas filas padrão do SQS=~
Incorreta. **O Amazon SQS Standard oferece alta taxa de transferência e entrega pelo menos uma vez, mas não é ideal para cenários que exigem processamento em tempo real e múltiplos consumidores lendo simultaneamente de um fluxo contínuo de dados**. Além disso, SQS não é um serviço de streaming, o que o torna inadequado para esse caso de uso.

~={red}Substituir o Kinesis Data Streams pelo Kinesis Data Firehose=~
Incorreta. **O Amazon Kinesis Data Firehose é um serviço totalmente gerenciado que facilita o carregamento confiável de dados em streaming para data lakes, armazenamentos e ferramentas analíticas.** **Ele é projetado para entrega automática e não permite que aplicações consumam diretamente os dados em tempo real**, diferentemente do Kinesis Data Streams. Portanto, não é adequado para casos que requerem múltiplos consumidores lendo dados simultaneamente em tempo real.

> **Kinesis Data Firehose:** Serviço de **entrega gerenciada de dados em lote**, voltado para ingestão contínua em destinos como Amazon S3, Redshift ou OpenSearch, **sem consumo em tempo real**.

##### Firehose X Data Stream
Uma empresa de investimentos deseja gerar continuamente análises de séries temporais das ações compradas por seus clientes. A empresa quer construir um ranking ao vivo com análises quase em tempo real dessas ações mais demandadas. **Qual das alternativas a seguir representa uma solução totalmente gerenciada com o menor custo para atender a esse caso de uso?**

~={green}Usar Kinesis Firehose para ingerir os dados e Kinesis Data Analytics para gerar as pontuações do ranking e as análises de séries temporais.=~
Correta. **O Amazon Kinesis Data Firehose é um serviço totalmente gerenciado que captura, transforma e entrega dados de streaming para destinos como Amazon S3, Redshift e outros, escalando automaticamente e sem necessidade de administração contínua**. **O Kinesis Data Analytics permite transformar e analisar dados em streaming em tempo real usando Apache Flink, com baixa latência e sem necessidade de gerenciar servidores**, sendo ideal para gerar rankings e análises de séries temporais quase em tempo real com custo otimizado.

~={red}Usar Kinesis Data Streams para ingerir os dados e Kinesis Data Analytics para gerar as pontuações do ranking e as análises de séries temporais.=~
Incorreta. Embora o Kinesis Data Streams seja altamente escalável e adequado para ingestão de grandes volumes de dados, **ele envolve custos adicionais relacionados à provisão e manutenção de shards.** **Para um cenário que prioriza uma solução totalmente gerenciada e de menor custo, o Kinesis Firehose é mais indicado, pois elimina a necessidade de gerenciamento de shards e reduz os custos operacionais.**

##### Função !FindInMap
Você está criando um template do CloudFormation para implantar sua aplicação CMS rodando em uma instância EC2 dentro da sua conta AWS. Como a aplicação será implantada em múltiplas regiões, você precisa **criar um mapa com todos os possíveis valores da AMI base**. Como você deve invocar a função !FindInMap para atender a esse caso de uso?

~={green}!FindInMap [ MapName, TopLevelKey, SecondLevelKey ]=~
Correta. A função Fn::FindInMap retorna o valor correspondente às duas chaves fornecidas dentro de um mapa declarado na seção Mappings do template. O primeiro parâmetro é o nome lógico do mapa, o segundo é a chave de nível superior (por exemplo, a região), e o terceiro é a chave de segundo nível (por exemplo, o tipo de arquitetura da AMI). Essa é a forma correta para acessar valores em mapas aninhados no CloudFormation.

> A função Fn::FindInMap suporta apenas dois níveis de chaves para acessar valores em um mapa declarado na seção Mappings. Passar três níveis de chave excede o número de parâmetros aceitos e causará erro na execução do template.
### AWS CLI
Você deseja paginar os resultados de um comando List do S3 para mostrar **100 resultados por página aos seus usuários** e minimizar o número de chamadas de API utilizadas. Quais opções da CLI você deve usar? (Selecione duas.)

~={green}--max-items=~
Correta. **A opção --max-items limita o número total de itens retornados pela chamada da CLI**, permitindo controlar quantos resultados são exibidos ao usuário, reduzindo a quantidade total de dados processados.

~={green}--starting-token=~
Correta. **--starting-token é usado para indicar o ponto de início para a paginação, permitindo que você continue a partir de um token específico e assim controle a navegação entre páginas de resultados.**

~={red}--page-size=~
Incorreta. **--page-size define quantos itens são processados por chamada interna da API, mas não controla a quantidade total de resultados exibidos ao usuário.** Por isso, não atende diretamente à necessidade descrita na pergunta.

~={red}--limit=~
Incorreta. --limit não é uma opção válida na AWS CLI para paginação e foi adicionada como alternativa incorreta.

Imagine uma aplicação web que lista **arquivos armazenados em um bucket S3**. Um desenvolvedor pode usar o comando abaixo para listar **100 objetos por página**:
1. aws s3api list-objects --bucket meu-bucket --max-items 100
O comando retorna os 100 primeiros itens e, junto com eles, um `**NextToken**`. Para exibir a próxima página, basta reutilizar esse token:
2. aws s3api list-objects --bucket meu-bucket --starting-token <NextToken> --max-items 100
